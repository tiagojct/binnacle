# todo.txt Format Specification (with Binnacle extensions)

Binnacle reads and writes the standard [todo.txt](http://todotxt.org) format, with minor pragmatic extensions.

## Core Format (standard todo.txt)

Each line in the file is either a task, a blank line, or a comment.

### Task syntax

```
[x] [priority] [completion_date] [creation_date] <description> [projects] [contexts] [key:value]
```

| Token | Position | Required | Format | Example |
|---|---|---|---|---|
| `x` | Start of line | No | `x ` (x + space) | `x 2026-06-07 ...` |
| Priority | After `x ` or start | No | `(A)` through `(Z)` | `(B) Buy milk` |
| Completion date | After `x ` | If completed | `YYYY-MM-DD` | `2026-06-07` |
| Creation date | After priority or completion date | Optional | `YYYY-MM-DD` | `2026-06-05` |
| Description | After dates | Yes | Free text | `Call Mom about Sunday` |
| Projects | Anywhere after description | No | `+projectname` | `+binnacle` |
| Contexts | Anywhere after description | No | `@contextname` | `@phone` |
| Key:value | Anywhere after description | No | `key:value` | `due:2026-06-15` |

### Edge cases

- A line with only whitespace is treated as blank (preserved on write).
- A line starting with `#` is treated as a comment. The parser reads it but does not interpret it as a task.
- `x` is only interpreted as completion when it is the very first character, followed by a space. `x-ray` in the description is not a completion marker.
- Priority must be uppercase `(A)` through `(Z)`. Lowercase `(a)` is treated as description text.
- Dates must match `YYYY-MM-DD` and be valid calendar dates (month 01–12, day valid for month).
- `key:value` pairs: key is lowercase alphanumeric + hyphens/underscores. Value is any non-space text. No quoting.

### Examples (valid)

```
(A) Call Mom @phone due:2026-06-08
This task has no priority and no date
x 2026-06-07 2026-06-05 Buy milk @errands
(B) +binnacle Write the parser spec due:2026-06-15
x (A) 2026-06-07 This was priority A and is now done
Report TPS-142 +acme @work due:2026-07-01
```

### Examples (invalid — treated as plain description)

```
(a) lower-case priority is not a priority
xBuy milk (no space after x)
```

## Binnacle extensions

These are pragmatic additions that do not break standard todo.txt compatibility. A file written by Binnacle remains fully parseable by any todo.txt-compliant tool.

### Extension 1: `done:` key

When a task is completed, Binnacle appends `done:YYYY-MM-DD` for readability:

```
x 2026-06-07 2026-06-05 Buy milk @errands done:2026-06-07
```

Standard todo.txt embeds the completion date at position 2 (`x YYYY-MM-DD`). Binnacle also adds `done:YYYY-MM-DD` so that completion metadata survives reordering and editing. Other tools will treat `done:2026-06-07` as an opaque key:value pair (harmless).

### Extension 2: `id:` key

Binnacle assigns a short, stable identifier to every task on first write:

```
(A) Call Mom @phone due:2026-06-08 id:a1b2
```

- Format: 4 characters, base62 `[0-9a-zA-Z]`
- Generated by Binnacle when a new task line is written
- Stable across reordering and edits
- Used internally for focus navigation and undo history
- Other todo.txt tools treat `id:a1b2` as an opaque key:value pair

### Extension 3: Preservation of ordering

Standard todo.txt does not guarantee that tools preserve line order. Binnacle preserves the exact line order of the file on read and write. If a task is completed, it is moved to the bottom of the file by default (configurable: `binnacle_keep_done_inline: true` in a comment at the top of the file).

### Extension 4: File-level metadata

Binnacle reads comment lines at the top of the file as configuration:

```
# binnacle_keep_done_inline: true
# binnacle_default_view: focus
```

| Key | Values | Default | Description |
|---|---|---|---|
| `binnacle_keep_done_inline` | `true` / `false` | `false` | If true, completed tasks stay in place |
| `binnacle_default_view` | `list` / `focus` | `list` | Default view on open |

Unknown `binnacle_*` keys are ignored. Comments without `binnacle_` prefix are treated as user notes and preserved.

## Parser rules (implementation order)

1. Trim trailing whitespace from the line; skip if blank.
2. If line starts with `#`, treat as comment.
3. Check completion: if line matches `^x\s`, mark `done = true` and strip the `x ` prefix.
4. Extract completion date if present: `^\d{4}-\d{2}-\d{2}\s` after the `x `.
5. Check priority: if remaining string starts with `^\([A-Z]\)\s`, capture priority and strip.
6. Extract creation date if present: `^\d{4}-\d{2}-\d{2}\s`.
7. The remaining string is the description.
8. Scan description for `@\S+` (contexts), `\+\S+` (projects), `\S+:\S+` (key:value).
9. Normalise: sort contexts, projects, and key:value pairs alphabetically on write.

## Write rules

When Binnacle serialises a task back to a line:

```
[x] [completion_date] [priority] [creation_date] <description> [contexts] [projects] [key:values]
```

- If `done = true`: start with `x completion_date` (and `priority` if present before completion).
- If `done = false` and `priority` is set: `(A) ` prefix.
- If `creation_date` is set and task is not done: date after priority.
- Description: as entered by user, trimmed.
- Append `@contexts` alphabetically, then `+projects` alphabetically, then `key:value` pairs alphabetically.
- Binnacle-specific keys (`done:`, `id:`) are included in the key:value section but are never exposed in the UI.
